# 记忆 Flush Protocol

**创建时间**: 2026-03-25
**用途**: 防止上下文窗口填满，确保重要信息不丢失

---

## 📊 上下文窗口监控

### 监控指标
- **当前使用率**: 每次重要操作后检查
- **阈值设置**:
  - < 50%: 正常操作
  - 50-70%: 增加警惕，写关键点到文件
  - 70-85%: 主动刷新，写入所有重要信息
  - > 85%: 紧急刷新，完整上下文总结

### 检查时机
1. 完成重要任务后
2. 长时间对话后（> 20 轮）
3. 接收大量数据后
4. 执行复杂操作前

---

## 🔄 Flush 触发条件

### 自动触发
- 上下文使用率 > 85%
- 单次对话 > 30 轮
- 接收 > 100KB 数据

### 手动触发
- 用户明确要求"记下来"
- 发现重要决策或配置
- 执行关键操作

---

## 📝 Flush 内容规范

### 必须写入的信息
1. **决策**: 重要的决策和理由
2. **配置**: 系统配置和参数变更
3. **问题**: 遇到的问题和解决方案
4. **学习**: 新学到的技能和知识
5. **任务**: 进行中的任务状态

### 写入文件选择
- **MEMORY.md**: 长期保留的核心信息
- **memory/YYYY-MM-DD.md**: 每日工作日志
- **memory/tasks.md**: 待办任务跟踪
- **memory/skills_kb.md**: 技能知识库
- 专门的专题文件

---

## 🎯 Flush 执行步骤

### Step 1: 检查上下文
```
session_status  # 查看当前使用率
```

### Step 2: 评估信息重要性
- 📌 关键: 必须写入 MEMORY.md
- 📝 重要: 写入 daily memory
- 📋 任务: 写入 tasks.md
- 🗑️ 临时: 可以忽略

### Step 3: 写入对应文件
```
# 关键决策 → MEMORY.md
# 工作日志 → memory/YYYY-MM-DD.md
# 任务状态 → memory/tasks.md
```

### Step 4: 确认写入
- 检查文件是否更新
- 验证内容是否完整
- 记录 Flush 时间

---

## ⚡ 紧急 Flush 流程（使用率 > 85%）

### 1. 立即停止新操作
```
⚠️ 上下文即将填满，正在保存信息...
```

### 2. 快速摘要
```markdown
## 当前状态
- 正在执行的任务
- 已完成的操作
- 待处理的请求
- 重要决策和配置
```

### 3. 写入文件
- 保存到 memory/emergency_flush_YYYYMMDD_HHMMSS.md
- 标注"紧急"

### 4. 通知用户
```
⚠️ 上下文已保存，继续执行任务...
```
